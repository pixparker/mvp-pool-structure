# Deploy flows

Two transports for getting a built image from your laptop to the pool VPS. Both produce the same end state (image present locally on the server, with matching tag), so the per-MVP `compose.yaml` works for either — only the *transport* differs.

## Registry mode (default)

```
[laptop]                                                      [pool VPS]
docker buildx build  →  docker push  →  Caddy + TLS + basic_auth  →  registry:2  ─┐
                                                                                   │
                                                              docker compose pull  ←┘
                                                              docker compose up -d
```

- Operator: `mvpool-local deploy <slug> --from <path>` (no `--mode` needed; default).
- Self-hosted `registry:2` listens on `127.0.0.1:5000`; Caddy publishes it at `registry.${POOL_DOMAIN}` with the htpasswd generated by `bootstrap.sh`.
- Layered, so iterative deploys are tiny — only changed layers transfer.
- Clean rollback: every prior tag stays in the registry until you garbage-collect. Roll back via `mvpool rollback <slug> <tag>`.

### Auth

`bootstrap.sh` writes `deploy/infra/registry-auth/htpasswd` and prints the password once. To log in from your laptop:

```bash
mvpool-local login            # docker login registry.${POOL_DOMAIN}
```

The user defaults to `mvpool` (override with `MVPOOL_REGISTRY_USER` env or in `~/.config/mvpool/config`).

### Storage / GC

The registry stores images in the `registry_data` Docker volume. If you want to delete old tags:

```bash
# delete unreferenced manifests on the server
docker compose -f /srv/infra/compose.yaml exec registry \
  bin/registry garbage-collect /etc/docker/registry/config.yml
```

A retention policy (e.g., "keep last 10 tags per slug") can be a future cron job.

## Tarball mode (`--mode tarball`)

```
[laptop]                                                      [pool VPS]
docker buildx build  →  docker save | gzip  →  ssh "gzip -dc | docker load"  →  docker images
                                                                                docker compose up -d
```

- Operator: `mvpool-local deploy <slug> --from <path> --mode tarball`.
- No registry needed. Works when the server has no outbound network access for `docker pull`.
- The image is saved with its full registry-prefixed tag (`registry.<pool>/<slug>-web:<sha>`), so the same `compose.yaml` finds it locally.

### Tradeoffs

- Full image transfers every time — no layer dedup.
- No deploy history in any registry; rollback only works for tags still cached locally on the server.
- Best for: air-gapped servers, one-off MVPs, or temporary use while the registry is unreachable.

## Choosing

| You want…                                                 | Use         |
| ---------------------------------------------------------- | ----------- |
| Fastest iterative deploys                                 | registry    |
| Clean rollback to any past tag                            | registry    |
| Server can't reach outbound HTTPS                         | tarball     |
| Server can't accept inbound `docker push` (e.g. no DNS)   | tarball     |
| First deploy ever, registry isn't up yet                  | tarball     |

## Rollback

Both modes use the same rollback (the tag must exist on the server):

```bash
mvpool-local rollback <slug> <prior-tag>
# = mvpool deploy <slug> --tag <prior-tag>
```

In registry mode, this triggers `docker compose pull` for the prior tag (which hits the registry, finds the manifest, and pulls only any missing layers). In tarball mode, the prior tag must still be present in the local Docker daemon on the server.

## Auditing deploys

Every successful deploy appends a line to `/srv/infra/state/deployments.jsonl`:

```json
{"ts":"2026-04-28T10:21:14Z","slug":"gamification","tag":"a1b2c3d","mode":"registry"}
```

`mvpool status` shows the last 10. Tail the file for more:

```bash
mvpool-local status
# or
ssh user@pool-vps "tail -n 50 /srv/infra/state/deployments.jsonl"
```
